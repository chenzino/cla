Agentic Coding Workflows

A comprehensive guide to setting up hierarchical multi-agent workflows in Claude Code. Learn how to orchestrate specialized subagents, parallelize work, and build production-grade agentic systems that scale across complex codebases.

◆

Architecture Overview

▼

Claude Code uses a hierarchical agent architecture where a main orchestrator agent delegates specialized tasks to independent subagents. Each subagent runs in its own context window, receives focused instructions, and reports results back to the parent.

┌───────────────────────────────────┐ │ Main Orchestrator Agent │ │ (plans, delegates, reconciles) │ └───────┬─────────┬─────────┬───────┘ │ │ │ ┌─────┴───┐ ┌───┴────┐ ┌───┴─────┐ │ Explore │ │ Plan │ │ General │ │ Agent │ │ Agent │ │ Purpose │ └─────────┘ └────────┘ └──────────┘

This architecture solves the context window problem: instead of one agent trying to hold an entire codebase in memory, each subagent maintains context only for its specific domain, leading to better focus and more accurate results.

How It Works

  1. The main agent receives your request and creates a plan
  2. It identifies subtasks that can be delegated or parallelized
  3. It spawns subagents via the Task tool with specific instructions
  4. Each subagent executes independently within defined boundaries
  5. Results flow back to the main agent, which reconciles outputs and resolves conflicts
  6. The main agent can spawn additional subagents if needed
ⓘ Each subagent is a semi-independent process with its own context window. It receives its assignment along with relevant context: which files to focus on, what output is expected, and any constraints to follow.

Built-in Subagent Types

▼

Claude Code ships with several specialized subagent types, each optimized for different kinds of work. Choose the right agent for the task to get the best results.

● Explore Agent

Fast codebase exploration using Haiku. Finds files by patterns, searches for keywords, and answers structural questions. Use with thoroughness levels: "quick", "medium", or "very thorough". Read-only — cannot edit files.

● Plan Agent

Software architect agent for designing implementation plans. Returns step-by-step plans, identifies critical files, and considers architectural trade-offs. Read-only — cannot edit files.

● General-Purpose Agent

Full-capability agent for complex, multi-step tasks. Has access to all tools including file editing, bash, web search, and more. Use for tasks that require real changes.

● Custom Agents

Define your own specialized agents as Markdown files with YAML frontmatter. Store at ~/.claude/agents/ (global) or .claude/agents/ (project).

Spawning a Subagent with the Task Tool

// The main agent uses the Task tool to spawn subagents // Each call specifies the agent type and a detailed prompt Task( subagent_type: "Explore", description: "Find auth endpoints", prompt: "Search the codebase for all authentication-related API endpoints in src/routes/ and src/api/" ) Task( subagent_type: "general-purpose", description: "Refactor auth module", prompt: "Refactor the authentication middleware to use JWT tokens instead of session cookies." )
⚠ Cost awareness: Each subagent consumes tokens independently. Running multiple agents in parallel increases usage. Limit to 3–4 concurrent subagents for most work.

Configuring CLAUDE.md for Agent Workflows

▼

CLAUDE.md files provide persistent instructions and context to Claude Code. They are loaded hierarchically and are essential for configuring how agents behave across your project.

Hierarchy of CLAUDE.md Files

# Files are loaded in this order (later overrides earlier) ~/.claude/CLAUDE.md # Global — applies to all projects ./CLAUDE.md # Project root — shared with team ./src/CLAUDE.md # Directory-level — scoped context ./src/api/CLAUDE.md # Deeper nesting for specific areas

What to Put in CLAUDE.md

  • Project conventions — coding style, naming patterns, test framework preferences
  • Architecture notes — how the system is structured, key design decisions
  • Agent instructions — when to spawn subagents, preferred delegation patterns
  • Build & test commands — how to run tests, lint, and build the project
  • Boundaries — files or directories agents should not modify

Example Project CLAUDE.md

# Project: My API Server ## Architecture Express.js REST API with PostgreSQL. Routes in src/routes/, models in src/models/, middleware in src/middleware/. ## Testing Run tests with `npm test`. Integration tests require `docker-compose up -d db` first. ## Agent Guidelines - Use Explore agents for codebase discovery before changes - Use Plan agents for any task touching 3+ files - Always run tests after making changes - Never modify files in src/generated/
ⓘ The better your CLAUDE.md files, the better Claude Code performs. Anthropic's own teams report that well-crafted CLAUDE.md files make Claude Code "excel at routine tasks like setting up new data pipelines."

Creating Custom Subagents

▼

Beyond the built-in types, you can define custom subagents as Markdown files with YAML frontmatter. They are automatically available when Claude Code detects a matching task.

Agent File Locations & Priority

# Priority (highest to lowest): --agents CLI flag # Current session only .claude/agents/ # Project-level (shared with team) ~/.claude/agents/ # User-level (all projects) Plugin agents/ directory # Where plugin is enabled

Example: Security Audit Agent

--- name: security-auditor description: Reviews code for security vulnerabilities including OWASP Top 10 concerns tools: - Read - Grep - Glob - WebSearch --- # Security Audit Agent You are a security-focused code reviewer. When analyzing: 1. Check for SQL injection, XSS, CSRF vulnerabilities 2. Review authentication and authorization logic 3. Look for hardcoded secrets or credentials 4. Verify input validation at system boundaries 5. Check dependencies for known CVEs Report with severity: CRITICAL, HIGH, MEDIUM, LOW. Reference specific file and line number.

Restricting Subagent Spawning

# A coordinator that can only spawn specific agents --- name: coordinator description: Coordinates work across specialized agents tools: - Task(worker, researcher) # Only these agents - Read - Bash --- # Note: subagents cannot spawn other subagents # Omit Task from tools list to prevent spawning entirely

Parallelizing Agent Work

▼

One of the most powerful patterns in Claude Code is running multiple subagents in parallel. When tasks are independent, launching them simultaneously dramatically reduces total completion time.

When to Parallelize

  • Independent research — searching different parts of the codebase simultaneously
  • Non-overlapping edits — modifying files that don't depend on each other
  • Multi-concern analysis — security + performance + style checks at once
  • Test + lint — running verification steps concurrently

Parallel Task Pattern

// Launch multiple independent agents simultaneously // The main agent sends all Task calls in a single message Task(subagent_type: "Explore", prompt: "Map all REST endpoints in src/routes/") Task(subagent_type: "Explore", prompt: "Catalog all database models and relationships") Task(subagent_type: "Plan", prompt: "Design a migration plan from REST to GraphQL")

Background Agents

For long-running tasks, spawn agents in the background so the main agent can continue other work.

Task( subagent_type: "general-purpose", run_in_background: true, prompt: "Run the complete test suite" ) // Main agent continues immediately // Gets notified when tests finish

Isolated Worktrees

For changes that might conflict, use git worktrees to give each agent an isolated copy of the repository.

Task( subagent_type: "general-purpose", isolation: "worktree", prompt: "Refactor the auth module to use OAuth2" ) // Agent gets its own copy of the code // Changes only merge if tests pass

When NOT to Parallelize

  • When Task B depends on output from Task A
  • When multiple agents would edit the same files
  • When you need to validate results sequentially

Hooks & MCP Server Integration

▼

Hooks and MCP (Model Context Protocol) servers extend agent workflows with custom automation and external tool access.

Hooks

Shell commands that execute in response to agent events: PreToolUse, PostToolUse, Stop, SubagentStart, SubagentStop, TaskCompleted.

// In settings.json or .claude/settings.json { "hooks": { "PreToolUse": [{ "matcher": "Bash", "hooks": [{ "type": "command", "command": "echo 'Bash invoked' >> agent.log" }] }], "PostToolUse": [{ "matcher": "Write", "hooks": [{ "type": "command", "command": "npm run lint --fix" }] }] } } // Exit codes: 0 = allow, 2 = block the operation // Keep hooks under 500ms to avoid degrading sessions

MCP Servers

MCP servers provide agents with access to external tools and data sources. They are available to all agents in the session, including subagents.

// In settings.json { "mcpServers": { "database": { "command": "npx", "args": ["@modelcontextprotocol/server-postgres", "postgresql://localhost/mydb"] }, "github": { "command": "npx", "args": ["@modelcontextprotocol/server-github"], "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" } } } }
ⓘ MCP servers are shared across all agents in the session. Subagents can query databases, access GitHub, or use any MCP tool without additional configuration. When tool descriptions exceed 10% of context, tool search activates automatically.

Building with the Claude Agent SDK

▼

For fully custom agent orchestration beyond what Claude Code provides natively, the Claude Agent SDK (Python & TypeScript) lets you build and deploy custom multi-agent systems programmatically.

Basic Agent SDK Example

import { Agent, tool } from "@anthropic-ai/claude-agent-sdk"; // Define a specialist agent const reviewer = new Agent({ name: "code-reviewer", model: "claude-sonnet-4-6", instructions: `Expert code reviewer. Focus on correctness, security, and performance.`, tools: [readFile, searchCode, analyzeAST] }); // Define an orchestrator that delegates const orchestrator = new Agent({ name: "lead-developer", model: "claude-opus-4-6", instructions: `Coordinate development tasks. Delegate reviews to the reviewer agent.`, tools: [reviewer.asTool(), editFile, runTests] }); // Run the orchestrator const result = await orchestrator.run( "Review and fix the auth module security issues" );
ⓘ Use Opus for orchestrator agents needing strong reasoning. Use Sonnet for focused specialist tasks. Use Haiku for simple, high-volume tasks like file searches. The SDK supports auto-compaction when context nears ~95% capacity.

Real-World Workflow Patterns

▼

Pattern 1: Explore → Plan → Execute

The most common pattern. Understand the codebase, plan changes, then execute with confidence.

// Step 1: Explore (parallelize research) Task(subagent_type: "Explore", prompt: "Find all payment code") Task(subagent_type: "Explore", prompt: "Find all Stripe API calls") // Step 2: Plan (after exploration returns) Task(subagent_type: "Plan", prompt: "Design Stripe v2 to v3 migration") // Step 3: Execute (after plan approval) Task(subagent_type: "general-purpose", prompt: "Implement the plan...")

Pattern 2: Parallel Feature Development

Build independent components simultaneously across different layers.

// Three agents working on different parts at once Task(subagent_type: "general-purpose", prompt: "Build the database schema and models") Task(subagent_type: "general-purpose", prompt: "Build the REST API endpoints") Task(subagent_type: "general-purpose", prompt: "Build the React components")

Pattern 3: Review & Verify Pipeline

After code changes, run multiple verification agents in parallel.

// Verify changes from multiple angles at once Task(subagent_type: "general-purpose", prompt: "Run test suite") Task(subagent_type: "security-auditor", prompt: "Audit changed files") Task(subagent_type: "code-reviewer", prompt: "Review all changes")

Pattern 4: Agent Teams (Experimental)

For sustained parallel work, Agent Teams give each agent its own persistent context with peer-to-peer messaging via shared inbox files on disk.

// Enable Agent Teams (experimental) { "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" } } // Teams communicate via JSON inbox files: // ~/.claude/<teamName>/inboxes/<agentName>.json // Start with 3-5 teammates, ~5-6 tasks each

Best Practices & Tips

▼

Agent Management

  • Limit concurrency — 3–4 subagents max. More creates coordination overhead that outweighs benefits.
  • Be specific in prompts — tell agents exactly which files, directories, or patterns to focus on.
  • Use the right type — Explore for reading, Plan for designing, General-Purpose for executing.
  • Don't duplicate work — if you delegate to a subagent, don't also perform the same searches yourself.

Context Window Management

  • Offload to subagents — protect the main context window from excessive search results.
  • Summarize, don't dump — have subagents return concise summaries, not raw file contents.
  • Resume, don't restart — agents can be resumed by ID, preserving their full prior context.
  • Use persistent memory — set memory: user in agent YAML to build institutional knowledge across sessions.

Reliability

  • Always verify — run tests after agents make changes.
  • Use CLAUDE.md guardrails — specify files and directories agents should never modify.
  • Start simple — begin with single-agent workflows, add subagents as complexity demands.
  • Leave code clean — each session should make incremental single-feature progress and leave the codebase in a mergeable state.

Settings Hierarchy

# Settings loaded with increasing precedence: ~/.claude/settings.json # User (lowest) .claude/settings.json # Project (shared) .claude/settings.local.json # Project (personal) Enterprise managed policies # Org-level (highest) # Run /status inside Claude Code to see active sources
⚠ Start small: For most day-to-day work, stick with standard Claude Code and reserve multi-agent workflows for architecture reviews, security audits, or complex multi-file refactors. Three focused agents outperform five scattered ones.
